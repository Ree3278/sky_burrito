# Sky Burrito RL 系统框架总结

**更新日期:** 2026年4月19日  
**项目:** sky_burrito  
**语言:** 中文 (无emoji)

## 环境概述

Sky Burrito RL系统是一个基于Gymnasium的多智能体无人机队伍优化环境，用于学习最优的送货路线和位置调度策略。该系统通过强化学习算法(PPO)来优化无人机的任务分配和位置调度，目标是最大化订单履行率同时最小化排队时间。

---

## 状态空间 (State Space)

### 观察向量组成

每个环境步骤返回的观察包含以下信息:

#### 1. 无人机状态 (每架无人机)
- 当前位置 (hub_id): 0-8共9个枢纽编号
- 电池电量 (SOC): 0.0-1.0归一化值
- 是否正在送货 (delivering): 布尔值
- 剩余送货时间 (steps_remaining_delivery): 分钟数

#### 2. 枢纽状态 (每个枢纽)
- 等待中的订单数 (queue_length): 非负整数
- 空闲无人机数 (idle_drones): 可用无人机数量
- 需要充电的无人机数 (charging_drones): 充电中的无人机数

#### 3. 全局信息
- 当前小时 (hour): 0-23
- 当前分钟 (minute): 0-59
- 总排队数 (total_queue_backlog): 全网络订单总数

### 观察形状和特征
- 总维度: 多个特征维度，通常100+维
- 对于9个枢纽的完整网络: 约100+ 维特征向量
- 数据类型: numpy float32数组
- 特征被归一化到相似的范围以便学习

## 动作空间 (Action Space)

### 动作设计

动作空间是一个9维的离散空间，每个维度代表一个枢纽的订单执行数量。

```
动作向量 = [hub_0_action, hub_1_action, ..., hub_8_action]
```

每个hub_i_action 的含义:
- 0: 该枢纽不分配无人机执行新订单
- 1-30: 在该枢纽分配最多1到30架无人机来执行等待的订单

### 动作执行逻辑

对于每个枢纽的动作值:

1. 若值为0: 该枢纽不进行任何订单分配
2. 若值为1-30: 尝试在该枢纽分配该数量的无人机来执行订单
3. 实际执行数 = min(请求分配数, 可用空闲无人机数, 排队订单数)

这种设计确保了:
- 无人机数量不会超过当前可用数量
- 订单数量不会超过队列中的实际订单
- 智能体学会在资源约束下的最优分配

## 奖励函数 (Reward Function)

### 多目标奖励设计

总奖励由五个主要组件组成，每个组件优化不同的目标:

#### 1. 履行奖励 (Fulfillment Bonus)
```
履行奖励 = 已完成订单数 x 100
```
特点:
- 每完成一个订单获得100分的奖励
- 直接激励系统快速完成订单
- 是唯一的正奖励来源之一

#### 2. 队列积压惩罚 (Queue Backlog Penalty)
```
队列惩罚 = -当前排队订单总数 x 144
```
特点:
- 每个排队订单每个时间步扣144分
- 主要负奖励来源，典型值占总奖励的99%以上
- 强制系统快速减少未处理的订单积压
- 当前最佳模型 (30无人机) 的典型值: -132637

#### 3. 起重惩罚 (Craning Penalty)
```
起重惩罚 = -起重事件数 x 10000
```
特点:
- 每次起重操作扣10000分
- 当无人机无法立即执行分配的订单时触发
- 在优化良好的解中很少发生 (通常0)
- 成本高昂的操作

#### 4. 闲置无人机奖励 (Idle Bonus)
```
闲置奖励 = 平均空闲无人机数 x 1
```
特点:
- 保持无人机可用性的小额奖励
- 防止系统让所有无人机同时工作
- 贡献很小，通常约+25
- 有利于应对突发订单

#### 5. 航程惩罚 (Deadhead Penalty)
```
航程惩罚 = -总航程距离
```
特点:
- 当前优化中未触发 (设为0)
- 为未来更精细的优化保留
- 可用于最小化无空载飞行

### 总奖励计算
```
总奖励 = 履行奖励 + 队列惩罚 + 起重惩罚 + 闲置奖励 + 航程惩罚
       = 履行奖励 + (主导项:队列惩罚) + 其他项
```

### 奖励特性
- 典型的总奖励范围: -180000 到 -120000 (对于30无人机配置)
- 队列惩罚主导整体分数，占比 >99%
- 系统设计使得减少排队积压是最重要的优化目标
- 负奖励的设计强制智能体在可行的情况下最大化履行

## 环境动态 (Environment Dynamics)

### 每个时间步的执行流程

1. 收集当前观察
   - 无人机位置、电池电量、送货状态
   - 每个枢纽的订单队列长度
   - 全局时间信息

2. 智能体返回动作
   - 9维向量表示每个枢纽的订单分配数

3. 执行动作
   - 分配空闲无人机到各枢纽的订单
   - 更新无人机的送货任务和剩余时间

4. 模拟时间进度
   - 时间前进1分钟
   - 已送货的无人机返回各自的起始枢纽
   - 返回的无人机开始充电
   - 新订单根据需求模型到达

5. 状态更新
   - 更新所有无人机的电池电量
   - 更新枢纽的队列状态
   - 计算当前时间步的奖励

### 时间刻度和剧集长度
- 每个环境步 = 1分钟模拟时间
- 第一阶段: 一个完整剧集 = 6小时 = 360步
- 第二阶段: 一个完整剧集 = 12小时 = 720步
- 第三阶段: 一个完整剧集 = 24小时 = 1440步

### 订单生成模型 (MGk排队理论)

基于M/G/k排队理论的需求生成:

#### 枢纽特性
- 每个枢纽有唯一的到达率参数λ (lambda) 和服务变异系数c_s²
- 枢纽9 (最繁忙): λ=2.2, c_s²=0.5
- 枢纽3: λ=1.8, c_s²=0.8
- 枢纽11: λ=1.6, c_s²=0.6

#### 时间依赖性
- 基于当前小时的需求变化
- 早餐高峰 (7-9 AM): 高需求
- 午餐高峰 (11 AM-1 PM): 高需求
- 晚餐高峰 (6-8 PM): 高需求
- 夜间 (10 PM-6 AM): 低需求

#### 订单到达
- 使用泊松过程生成订单
- 每个时间步根据当前λ值生成到达数
- 典型的每天订单数: 1800-2000个订单

## 环境配置参数 (Configuration)

### 核心参数

| 参数 | 值 | 说明 |
|-----|----|----|
| 活跃枢纽数 | 9个 | Hub 0, 1, 2, 3, 5, 6, 7, 9, 10, 11 |
| 无人机数量 | 10, 20, 30, 40, 50 | 不同运力配置 |
| 每个订单飞行时间 | 12分钟 | 最坏情况飞行时间 |
| 充电时间 | 2分钟 | 返回后的电池恢复时间 |
| 电池容量 | 1.0 (归一化) | 满电状态 |
| 每分钟耗电 (飞行) | 0.002 | 12分钟航程约消耗2.4% |
| 每分钟耗电 (待机) | 0.00002 | 非常低的待机消耗 |

### 三个渐进式训练阶段

#### 第一阶段 (Phase 1) - 单枢纽学习
- 只有1个活跃枢纽
- 无人机数: 20架
- 剧集长度: 6小时
- 总训练步数: 50000步
- 目标: 学习基本的订单-无人机匹配

#### 第二阶段 (Phase 2) - 双枢纽协调
- 有2个活跃枢纽 (双向连接)
- 无人机数: 20架
- 剧集长度: 12小时
- 总训练步数: 100000步
- 目标: 学习枢纽间的无人机转移策略

#### 第三阶段 (Phase 3) - 完整网络优化
- 所有9个枢纽都活跃
- 无人机数: 10, 20, 30, 40, 50 (多个配置)
- 剧集长度: 24小时
- 总训练步数: 每个配置150000步
- 目标: 学习全网络的全天优化策略

## 智能体配置 (PPO Agent Config)

### Proximal Policy Optimization (PPO) 设置

| 参数 | 值 | 说明 |
|-----|----|----|
| 算法 | PPO (Proximal Policy Optimization) | 策略梯度算法 |
| 网络架构 | MlpPolicy (多层感知机) | 两层隐藏层 |
| 隐藏层大小 | 64, 64 | 每层64个神经元 |
| 激活函数 | ReLU | 整流线性单元 |
| 学习率 | 初始0.0003 | 可根据阶段调整 |
| 批量大小 | 64 | 每次策略更新的样本数 |
| PPO裁剪范围 | 0.2 | 防止策略更新过大 (epsilon=0.2) |
| 熵系数 | 0.01 | 鼓励探索，防止过早收敛 |
| 折扣因子 | 0.99 | 未来回报的权重 |
| GAE λ | 0.95 | 广义优势估计的平滑参数 |
| 最大梯度范数 | 0.5 | 梯度剪切以稳定训练 |

### 训练细节

| 参数 | 值 | 说明 |
|-----|----|----|
| 优化器 | Adam | 自适应矩估计 |
| 价值网络系数 | 0.5 | 值函数损失权重 |
| 更新周期 | 4 | 每4步更新一次 |
| 小批量大小 | 32 | 单次梯度更新的批大小 |
| 梯度积累步数 | 1 | 无积累 |

## 主要问题分析 (Key Issues Analysis)

### 当前主要瓶颈

#### 问题1: 队列积压过大
- 平均921个订单在队列中等待
- 虽然履行率86%已经不错
- 但新订单到达速度快于完成速度
- 这是奖励为负的主要原因

#### 问题2: 无人机分配不够激进
- 平均只有8.9架无人机空闲
- 总共30架，约30%的空闲率
- 表明还有进一步优化的空间
- 可以尝试更激进的分配策略

#### 问题3: 队列管理策略欠佳
- 不仅是位置优化问题
- 还涉及流量平衡和动态预位置
- 需要提前预测需求并提前部署
- 当前策略缺乏前瞻性

### 问题根本原因
- 神经网络学到了基本的订单-无人机匹配
- 但未学到更复杂的动态规划策略
- 奖励信号强但可能需要更多的训练
- 或需要改变奖励函数的权衡

## 未来改进方向 (Future Improvements)

### 短期改进 (1-2周)

1. 奖励塑形 (Reward Shaping)
   - 调整各奖励组件的权重
   - 增加队列惩罚的权重
   - 添加早期完成的奖励

2. 更激进的再平衡逻辑
   - 提高无人机的分配主动性
   - 考虑超前部署
   - 基于预测的主动转移

3. 需求压力调整
   - 如果生成的需求过高，进行调整
   - 与实际运营数据对齐
   - 验证MGk参数的准确性

### 中期改进 (1-3个月)

1. 多智能体RL
   - 每架无人机作为独立智能体
   - 分散决策制定
   - 更可扩展的架构 (100+ 无人机)

2. 实时需求预测
   - 提前1-2小时预测需求
   - 基于历史模式学习
   - 提供给无人机进行主动布置

3. 改进的网络架构
   - 注意力机制处理依赖关系
   - 图神经网络表示枢纽拓扑
   - 递归网络处理时间序列

4. 实世界验证
   - 在实际Sky Burrito系统上测试
   - 比较模拟性能与实际性能
   - 迭代模型改进

### 长期改进 (3-6个月+)

1. 城市级规模优化
   - 从9个枢纽扩展到100+个
   - 多区域协调
   - 跨城市学习转移

2. 流动管理和预测
   - 实时需求预测
   - 动态定价
   - 供给-需求平衡

3. 异构智能体
   - 不同类型的无人机 (速度、容量、航程)
   - 每种类型的任务分配
   - 成本效益优化

4. 完整系统集成
   - 与订单管理系统集成
   - 与充电基础设施集成
   - 与实时监控集成

## 总结 (Summary)

### 系统架构总体特点

Sky Burrito RL系统是一个复杂的、多目标的优化环境，具有以下特点:

1. 状态空间
   - 多维度 (100+维)
   - 包含无人机微观状态和枢纽宏观状态
   - 具有强的时间依赖性

2. 动作空间
   - 9维离散空间
   - 每个维度0-30范围
   - 受资源约束的可行性验证

3. 奖励函数
   - 多目标设计 (5个组件)
   - 队列惩罚主导 (99%+)
   - 强烈激励订单完成和队列减少

4. 环境动态
   - 1分钟时间刻度
   - 基于MGk理论的现实需求
   - 多枢纽的网络结构

### 当前成果

- 30无人机配置达到86%的履行率
- 成功训练了PPO智能体
- 识别了主要瓶颈 (队列积压)
- 建立了基准和改进方向

### 关键发现

1. 排队积压是主要限制因素，而非无人机数量
2. 30无人机是成本-性能的最优点
3. 现有策略已学到基本规则，但缺乏前瞻性
4. 更激进的队列管理策略有很大改进空间

### 建议行动

短期: 实施奖励塑形和再平衡逻辑改进
中期: 探索多智能体方法和需求预测
长期: 实现完整系统集成和城市级规模
